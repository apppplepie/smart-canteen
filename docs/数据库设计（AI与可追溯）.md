# 数据库设计（AI 与可追溯）

> 本章体例说明：下文按学位论文常见写法组织，可采用「第 X 章」标题承接前文系统分析；表结构以 `backend/src/main/resources/scs.sql` 为权威依据。

---

## X.1 本章概述与研究范围

智慧食堂类信息系统具有业务域多、流程交叉明显的特点：既包含点餐、排队、评价等高频交易，又包含反馈内容、食安抽检与营养管理等监管与扩展需求；在引入大语言模型与检索增强能力后，还需在持久化层区分**在线业务数据**、**语义检索索引**与**审计类数据**，以保证系统可部署、可维护与可追溯。

本章从数据库角度阐述本系统的逻辑结构设计，主要研究范围包括：（1）面向多业务域的概念划分与逻辑表设计；（2）通过外键删除与更新策略实现的业务一致性控制；（3）面向食安场景的食材—菜品—订单—入库—检测链条建模；（4）在线事务数据与按日/按周期聚合的快照及报告类数据的层次划分；（5）AI 对话、向量检索与审计日志等智能增强相关表的设计要点。

本章最后给出各数据表的逻辑说明，作为系统实现的直接依据。

## X.2 面向智慧食堂的多域关系数据库概念模型与逻辑模型设计

### X.2.1 概念域划分

结合需求与业务流程，本系统将持久化对象划分为以下**概念域**，各域之间通过用户、商家等核心实体关联，形成多域关系模型。

（1）**用户与组织域**：以 `users` 为用户主实体，以 `vendors` 为食堂窗口/商家主实体，承载身份、角色与组织维度，为订单、内容、评价与 AI 会话提供统一引用。

（2）**交易与履约域**：包括 `menu_items`、`orders`、`order_items`、`queue_entries`、`call_events` 等，描述选品、下单、金额、排队状态及叫号事件，支撑「选餐—支付/下单—排队—取餐」闭环。

（3）**内容与互动域**：以 `posts` 为核心，辅以 `post_comments`、`post_likes`、`vendor_reviews` 等，承载反馈、食堂圈动态、评分与互动数据，为运营分析与语义检索提供文本来源。

（4）**食安与营养域**：包括 `materials`、`menu_item_materials`、`stock_movements`、`retained_samples`、`test_reports`、`nutrition_logs` 等，支撑食材主数据、配方关联、入库流水、留样与检测报告及营养记录。

（5）**智能与审计域**：包括 `ai_conversations`、`ai_messages`、`agent_requests`、`post_vector_index`、`audit_logs` 等，分别承担对话持久化、工具调用结构化存储、帖子向量索引及操作审计。

（6）**分析与快照域**：包括 `canteen_daily_snapshots`、`ai_period_reports` 等，用于存放按日或按周期的预聚合指标与 AI 周期报告，与在线业务表解耦。

上述划分在逻辑模型上体现为实体—联系设计：一对多关系（如用户与订单、商家与菜品）、多对多关系（如菜品与食材通过 `menu_item_materials` 解析），以及通过外键实现的参照完整性。

### X.2.2 逻辑模型设计要点

逻辑设计阶段遵循以下原则：其一，**高频查询路径优先建索引**，例如在订单、排队、帖子等表上对用户、商家、状态、时间字段建立组合或单列索引；其二，**易变宽列与索引数据分离**，例如帖子正文存于 `posts`，语义检索向量存于 `post_vector_index`，避免单表过度膨胀并便于重建索引；其三，**聚合结果适度冗余**，例如 `vendors` 中维护评分均值与人数，降低前台统计压力；其四，**扩展字段采用 JSON 适度承载**，如帖子多媒体、AI 工具调用明细等，在可查询性与模式灵活性之间折中。

## X.3 基于外键策略的业务一致性控制机制

关系数据库通过外键约束保证参照完整性；删除与更新行为需与业务规则一致，否则易出现孤儿数据或误删关键历史记录。本系统在外键定义中主要采用 `ON DELETE CASCADE`、`ON DELETE RESTRICT` 与 `ON DELETE SET NULL` 三类策略，其适用场景可归纳如下。

**（1）级联删除（CASCADE）**  
当子记录仅依附于父记录存在、父记录删除后子记录无独立业务含义时采用。例如：删除 AI 会话时，其下属消息应一并删除（`ai_messages` 参照 `ai_conversations`）；删除帖子时，评论与点赞应一并清理（`post_comments`、`post_likes` 参照 `posts`）；删除排队条目时，相关叫号事件应删除（`call_events` 参照 `queue_entries`）。该类设计可减少无效残留，降低数据清理成本。

**（2）限制删除（RESTRICT）**  
当父实体被其他业务强依赖、删除前必须先处理从属业务时采用。例如：存在关联订单时限制直接删除用户或商家（`orders` 对 `users`、`vendors`）；存在评价记录时限制随意删除被引用对象，以保证账务与合规数据不被静默破坏。`vendor_reviews` 对 `orders` 采用限制策略，保证「一订单一评价」约束与历史可追溯。

**（3）置空删除（SET NULL）**  
当子记录需要保留历史痕迹，但允许父实体标识解除时采用。例如：`order_items` 中菜品下架后仍保留订单明细，可将 `menu_item_id` 置空；`ai_conversations` 中用户注销或匿名化场景下，可将 `user_id` 置空以保留会话结构而弱化个人标识（具体以业务实现为准）。`lost_items` 中未登录用户寻物时 `user_id` 可为空，亦体现弱绑定设计。

通过上述场景化选择，本系统在库表层面将**业务规则**与**引用完整性**对齐，为后续应用层权限控制与审计提供一致的数据基础。

## X.4 食安事件的全链路可追溯数据模型设计与实现

食品安全与溯源类需求要求数据链能够回答：某次消费涉及的菜品由哪些食材构成，相关食材的入库与供应商信息如何追溯，抽检与留样、检测报告如何关联到组织维度。

本系统在逻辑上形成如下**追溯链路**：`materials` 维护食材主数据及过敏原等属性；`menu_item_materials` 建立菜品与食材的多对多关系，复合主键避免重复关联；`menu_items` 与 `order_items` 将消费行为落到具体菜品；`stock_movements` 记录入库与流转，并通过 `material_id` 与食材主数据关联（可与历史文本字段 `material` 并存以兼容迁移）；`retained_samples` 与 `test_reports` 分别支撑留样管理与检测结果存档，`nutrition_logs` 则将营养摄入与用户、订单关联，服务于健康分析而非单一溯源环节。

该设计的**重点**在于：将过敏原等强监管属性沉淀在食材层，避免仅在菜品描述中重复录入；将多对多关系显式表化，便于按食材反查菜品与订单；将检测报告与窗口（`vendor_id`）等组织维度关联，便于按责任主体统计分析。实现层面可在正文中结合典型用例（如疑似问题食材筛查）说明查询路径，以增强论文的工程完整性。

## X.5 事务型业务数据与统计快照的分层设计

在线业务处理（OLTP）要求低延迟写入与强一致性；经营看板、词云、周期分析报告等场景若以实时多表关联重算为代价，易在高峰期放大数据库负载。本系统将**事务型业务表**与**预聚合快照表、周期报告表**在模式上分离，形成层次化数据布局。

`canteen_daily_snapshots` 按自然日及作用域（如全局、窗口等）存储聚合指标与 JSON 形态的排行、词云等结果，并通过唯一约束保证同一日期、同一作用域仅保留一条快照记录，便于幂等更新与历史对比。`ai_period_reports` 存储 AI 生成的周期分析内容，包含报告类型、起止日期、作用域、生成状态、模型与提示版本等元数据，同样通过唯一约束避免重复生成；该类表与业务明细表**非强实时联动**，由定时任务或离线流程写入，论文中可明确其**最终一致性**定位，避免与 OLTP 实时性要求混淆。

该分层设计的意义在于：将读多写少的分析负载从核心交易路径中剥离，使数据库角色清晰，并为后续扩展数据仓库或消息驱动计算预留接口。

## X.6 智能增强与可解释性相关的数据模型

### X.6.1 双服务架构下的对话持久化

本系统智能体编排运行于独立 AI 服务，业务与对话持久化由应用服务写入 MySQL。`ai_conversations` 存储会话元数据，包括可选用户标识、标题、长对话摘要及摘要覆盖条数，并建立按用户与更新时间排序的索引，以支持会话列表查询；`ai_messages` 存储逐条消息，`role` 字段区分用户、助手、系统及工具消息，`tool_calls`、`tool_call_id`、`suggestions` 等以 JSON 形式保存结构化调用结果与快捷建议，便于前端渲染与事后解释。该设计使**模型交互过程**以结构化形式落库，而非仅存自然语言文本。

### X.6.2 检索增强（RAG）相关的索引存储

语义检索以业务帖子为对象：`posts` 提供标题与正文等业务字段；`post_vector_index` 以 `post_id` 唯一对应一篇帖子，存储参与嵌入的源文本、向量序列化结果及嵌入模型名称。帖子删除时向量行级联删除，保证索引与业务数据一致。检索流程可在应用层完成「向量化—相似度排序—回表取正文」，逻辑层保持**内容与向量索引分离**，便于更换嵌入模型并执行全量重建。

### X.6.3 审计与操作追溯

`audit_logs` 记录操作者、动作、对象类型与标识及 JSON 详情，用于关键行为审计。与 `ai_messages` 中的工具调用快照相结合，可从「业务操作日志」与「模型工具调用」两个层面支撑追溯需求，而无需在库内保存完整 HTTP 报文。

## X.7 数据表逻辑结构说明

以下按业务分组对各数据表作简要说明，每一项包含**业务用途**与**设计要点**，可与附录中的完整 DDL 对照。

### X.7.1 用户与基础主体

`users` 表存储用户账号、角色与基础资料，是订单、发帖、评价与 AI 会话的身份主表；`username` 唯一约束保证登录标识唯一，`role` 支撑分端权限，`is_deleted` 支持软删除以保留历史关联。`vendors` 表存储食堂窗口信息，作为菜单、订单、排队等模块的组织维度；`rating_avg` 与 `rating_count` 作为评分聚合缓存，`is_active` 支持上下线管理。

### X.7.2 点餐与排队

`menu_items` 维护可售菜品及价格、营养与制作时长，`vendor_id` 外键级联删除保证商家与菜品一致性，`is_available` 控制在售状态。`orders` 存储订单主记录，`user_id`、`vendor_id`、`status` 等字段配合索引支持多维查询。`order_items` 存储订单明细，`menu_item_id` 在菜品删除时可置空以保留历史成交价。`queue_entries` 存储排队号码与状态，`vendor_id` 与 `status` 组合索引支撑窗口看板。`call_events` 记录叫号事件，依附于排队条目级联删除，避免孤儿日志。

### X.7.3 社区与互动

`posts` 承载反馈与食堂圈内容，分层字段表达反馈类型、帖子类型与处理状态，`media_urls` 以 JSON 支持多媒体。`post_comments`、`post_likes` 分别存储评论与点赞，外键级联删除保证主帖删除时从属数据清理；`post_likes` 对 `(post_id, user_id)` 建唯一约束防止重复点赞。`vendor_reviews` 存储订单评价，`order_id` 唯一约束实现一单一次评价，并可与 `posts` 关联展示。

### X.7.4 食安、库存与营养

`materials` 为食材主数据表，`vendor_id` 可为空表示全局食材，过敏原等属性集中于食材层。`menu_item_materials` 为菜品与食材多对多表，复合主键防重复。`stock_movements` 记录出入库流水，`material_id` 关联食材主数据以强化追溯。`retained_samples` 管理留样，`sample_code` 唯一。`test_reports` 存储检测结果与报告链接，`nutrition_logs` 关联用户与订单记录营养摄入。

### X.7.5 AI、向量与周期报告

`agent_requests` 存储辅助性请求与响应记录，便于调试与回溯。`post_vector_index` 存储帖子向量索引，支撑语义检索。`ai_period_reports` 存储 AI 周期分析报告及生成元数据。前述 `ai_conversations`、`ai_messages`、`audit_logs` 见 X.6 节。

### X.7.6 运营统计与物联网

`canteen_daily_snapshots` 存储按日与作用域划分的经营与互动类聚合指标，与 OLTP 表解耦。`sensor_logs` 存储设备上报的时序指标，可按指标与时间查询，与食安分析形成多源数据补充。

### X.7.7 寻物拓展模块

`found_items` 与 `lost_items` 分别支撑招领与寻物；`lost_items` 允许 `user_id` 为空以兼顾匿名场景。`lost_item_comments` 存储寻物评论，与主帖级联删除。

## X.8 本章小结

本章围绕智慧食堂业务完成了多域关系数据库的概念划分与逻辑表设计，阐述了外键删除与更新策略在业务一致性控制中的场景化应用，给出了食安追溯链条与事务库、快照库分层的设计思路，并说明了 AI 对话、向量索引与审计相关表在可解释性与可追溯性方面的作用。全库表通过主键、外键、唯一约束与索引相互配合，在支撑日常业务处理的同时，为智能增强与监管分析类功能提供了结构化的数据基础。
