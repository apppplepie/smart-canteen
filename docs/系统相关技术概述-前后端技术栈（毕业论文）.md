## 2.1 前端开发技术概述

本系统前端采用 **前后端分离** 架构，浏览器通过 **HTTP/HTTPS** 调用 Spring Boot 提供的 **RESTful API**。为覆盖 **大屏展示、移动 H5、管理后台** 三类终端，仓库中实现了 **三个相互独立的前端工程**，构建工具与语言层面保持统一，以降低多端协作成本。

### 2.1.1 TypeScript

三端均选用 **TypeScript** 作为开发语言。其在 JavaScript 基础上增加了 **静态类型** 与 **接口/泛型** 等机制，能够在编译期发现部分错误，并为 **API 请求与响应** 提供类型约束。对于需要多人维护、接口字段较多的智慧食堂项目，TypeScript 有利于 **接口契约稳定** 与 **重构安全**，与后端 JSON 字段的演进可形成对照文档或共享类型定义（如 `shared/api` 中的类型与客户端封装）。

### 2.1.2 Vite

三端均使用 **Vite** 作为开发与构建工具。Vite 在开发阶段利用浏览器原生 ES Module 与按需编译，**冷启动与热更新** 较快；生产构建基于 Rollup，输出体积与兼容性可配置。相比传统打包器在大型项目上的全量打包开销，Vite 更贴合 **现代前端工程** 的节奏，且与 Vue、React 官方脚手架生态兼容良好。

### 2.1.3 React 技术栈（大屏与移动 H5）

**大屏子工程（`screen`）** 与 **移动 H5 子工程（`mobile`）** 采用 **React** 及其生态实现用户界面。

**技术要点**：函数式组件与 Hooks 组织状态与副作用；样式以 **Tailwind CSS** 等工具类方案为主，便于快速搭建 **数据展示与交互** 页面；数据访问通过 **`fetch` 或封装模块** 调用统一的后端 `/api` 前缀接口。

**选型考量**：大屏与移动端更强调 **信息布局、动效与交互流畅度**，React 在 **组件组合、社区组件与可视化相关实践** 上积累深厚，与 Vite + TypeScript 组合成熟稳定，适合作为 **面向师生用户的 C 端界面** 技术基座。

### 2.1.4 Vue 3 与 Element Plus（管理后台）

**管理后台子工程（`admin`）** 采用 **Vue 3** 与 **Element Plus**。

**技术要点**：Vue 3 的 **组合式 API** 与 **响应式系统** 便于组织复杂表单与表格状态；**Element Plus** 提供表格、表单、对话框、树形控件等 **中后台高频组件**，减少从零封装 UI 的工作量；路由与布局由 **Vue Router** 管理。

**与 React 端并存的原因（须在论文中写清）**：管理后台与大屏/移动端的 **产品形态不同**。后台以 **数据维护、权限控制、批量表格与表单** 为主，属于典型 **B 端中后台**；采用 **Vue 3 + Element Plus** 并 **基于成熟开源中后台模板进行二次开发**，可在 **导航布局、权限路由、请求拦截、主题与工程规范** 上复用社区方案，**缩短建设周期**，使开发精力集中于 **食堂业务字段与流程**，而不是重复搭建后台脚手架。大屏与移动端继续采用 React，可避免为统一框架而 **重写大量已稳定的 C 端页面**，在 **统一 TypeScript、统一 Vite、统一后端 API** 的前提下，形成 **「C 端 React + B 端 Vue」** 的务实组合。

**这样做的好处（可浓缩为条目）**：

1. **场景匹配**：React 侧重 C 端交互与展示；Vue 生态在 **企业级 UI 组件库** 上与中后台场景结合紧密。  
2. **工程效率**：管理端依托模板快速落地 **登录、布局、动态路由**，降低一次性成本。  
3. **协作边界清晰**：三端仅通过 **REST 契约** 与后端耦合，框架差异不扩散到服务端。  
4. **类型与构建统一**：TS + Vite 降低「双框架」带来的工具链分裂感。

### 2.1.5 Pinia（管理后台状态管理）

管理后台使用 **Pinia** 作为全局状态管理方案，用于 **用户信息、角色与权限路由** 等跨页面状态。相较于早期 Vuex 的模块样板代码，Pinia 的 API 更简洁，与 **组合式 API** 搭配自然，适合在 **登录后拉取权限、动态注册路由** 的流程中维护一致状态。

### 2.1.6 网络请求：Axios 与模块化 API

管理后台普遍通过 **Axios** 实例统一配置 **基址、超时、请求与响应拦截**（如 Token），减少重复代码。大屏与移动端可采用 **Axios 或 fetch**，以工程内实际封装为准；可将 **`shared/api`** 中对路径、类型与错误处理的抽取，概括为 **「多端共享 API 封装，减少重复与不一致」**。

### 2.1.7 Vue 与 React 的简要对比（论文可用表格）

| 维度 | Vue 3（本系统用于 admin） | React（本系统用于 screen / mobile） |
|------|---------------------------|-------------------------------------|
| 典型场景 | 中后台、表单表格、企业组件库 | C 端页面、复杂交互与生态组件 |
| 状态与逻辑 | 组合式 API、单文件组件直观 | Hooks 组合、函数式风格普遍 |
| UI 生态 | Element Plus 与后台模板成熟 | 社区方案多样，需更多自选与拼装 |
| 本系统用法 | 二次开发后台模板，专注业务 CRUD | 大屏与移动 UI 与数据展示 |

**结论句（可直接放小节末）**：二者并非互斥；本系统在 **统一语言与构建工具** 的前提下，按 **终端职责** 分别选型，属于 **工程上的合理权衡**，而非技术栈混乱。

## 2.2 后端开发技术概述

业务数据访问、事务一致性、与数据库及前端 **REST 契约** 的对接，由 **Java Spring Boot** 应用承担。智能化计算（大模型编排、工具调用、向量与语音等）由独立 **Python FastAPI** 服务承担。二者通过 **HTTP** 协作，浏览器 **不直连** AI 服务密钥所在端口，由 Java 侧转发与编排，形成清晰的 **安全边界与职责边界**。

### 2.2.1 Spring Boot 与 Spring Framework

**Spring Boot** 在 **Spring Framework** 之上提供了 **约定优于配置** 的启动方式、内嵌 Web 容器与 **起步依赖（Starter）** 管理，使应用能够快速以 **独立 JAR** 形式部署。本系统利用其 **依赖注入（IoC）** 与 **面向切面** 等能力，将控制器、服务、仓储等组件交由容器管理，降低模块间硬编码依赖，便于测试与替换实现。

### 2.2.2 Spring MVC 与 RESTful API

表现层采用 **Spring MVC** 的 **`@RestController`** 声明 **REST 风格** 接口：通过 **`@RequestMapping` 系列注解** 映射 URL 与 HTTP 方法，将请求体反序列化为 Java 对象，将返回值序列化为 **JSON**。该模式与前端 **前后端分离**、多端共享同一套 `/api` 资源路径的设计一致。

### 2.2.3 Spring Data JPA 与数据访问

持久化采用 **Spring Data JPA**，以 **实体（Entity）** 映射关系表，以 **Repository 接口** 声明查询方法，由框架生成实现，减少样板 **JDBC** 代码。对于订单、用户、帖子、会话等 **关系型、事务性** 较强的校园食堂业务，JPA 有利于 **对象模型与表结构对齐**，并在服务层保持 **事务边界** 清晰。

### 2.2.4 为何业务主后端选用 Spring Boot，而非 Node.js / NestJS / Next.js 等

论文中可能被问及：**为何不用 Node 做全栈 API，或使用 Next.js 一体化方案**。可从 **职责、生态与课程实践** 三方面作答（以下为建议表述，可按学院要求删减）。

**（1）职责与成熟度**  
本系统业务核心是 **关系型数据、事务与稳定 CRUD**，Java 与 **Spring 生态** 在企业与教学实践中长期用于该类系统；**Spring Data JPA**、事务管理与 **过滤器链** 等与「以数据库为中心的业务系统」匹配度高。**Node.js** 在 **高并发 IO、轻量 BFF** 场景表现突出，但将 **全部业务事务与复杂查询** 集中于 Node 并非必须；若团队已具备 **Java 与 Spring** 基础，选用 Spring Boot 可降低 **全栈语言切换** 带来的维护与教学验证成本。

**（2）与 Python AI 服务的分工**  
智能化能力依赖 **Python** 侧 **大模型、Agent、向量与语音** 等库；若再用 Node 实现第三套服务，会增加 **部署与监控维度**。当前方案为 **Java 单一业务网关 + Python 单一智能服务**，进程边界少、**故障定位** 简单。

**（3）Next.js 的定位**  
**Next.js** 以 **React 服务端渲染（SSR）与全栈路由** 见长，适合 **内容站、SEO 强需求或前后端同构** 的项目。本系统前端已为 **三个独立 SPA + REST**，后端以 **Spring Boot 统一 API** 为中心，**引入 Next 服务端** 会重叠职责，且与现有 **Vue 管理端** 并存时架构更难解释，故 **未采用 Next.js 作为业务后端**。

**（4）选型收益归纳**  
- **业务与智能解耦**：Java 守 **数据与事务**，Python 守 **模型与算法**。  
- **技术栈可解释**：答辩与文档中 **「Spring Boot + FastAPI」** 分工明确。  
- **安全**：敏感调用在服务端之间完成，符合 **B/S + 前后端分离** 的常见安全实践。

### 2.2.5 FastAPI 与 Spring Boot 的区别及在本系统中的角色

**FastAPI** 是基于 Python 的 **现代 Web 框架**，依托 **类型注解** 与 **Pydantic** 做请求校验，可自动生成 **OpenAPI** 文档，常与 **异步 IO** 结合处理高并发请求。其生态与 **PyTorch、LangChain、语音与向量** 等库同处 **Python 运行时**，迭代 Agent 与模型相关代码路径短。

**与 Spring Boot 的差异（论文可简表呈现）**：

| 维度 | Spring Boot（本系统业务层） | FastAPI（本系统智能服务层） |
|------|-----------------------------|-----------------------------|
| 运行时 | JVM | Python |
| 典型优势 | 企业级集成、JPA 事务、成熟运维 | 轻量 API、异步、与 ML/LLM 生态一体 |
| 本系统职责 | REST、会话落库、转发 AI、业务表 | 对话编排、工具调用、嵌入、语音等 |

**为何不把业务也全部迁入 FastAPI**：Python 同样能写 CRUD，但本系统将 **权威业务库、会话持久化与统一 `/api` 入口** 放在 Spring Boot，可充分利用 **JPA 与 Java 类型体系** 管理复杂实体关系，并使 **前端只面对一套 Java 暴露的 API**；FastAPI 专注 **算力与智能逻辑**，避免 **单一体量过大** 的 Python 单体难以按层治理的问题。该模式在业界常称为 **BFF / 网关层 + 领域服务** 思路的一种实现。

### 2.2.6 为何 Python 智能服务选用 FastAPI 而非 Flask

在 Python Web 开发中，**Flask** 是出现早、资料多、社区庞大的 **微框架（Microframework）** 代表：核心极简，路由与模板即可搭建服务，其余能力通过 **扩展（Extensions）** 按需拼装，灵活度高，常被视为「经典」入门与小型服务首选。

本系统智能子系统仍选用 **FastAPI**，主要基于 **智能服务场景的工程特征**，而非否定 Flask：

**（1）请求体验证与契约**  
对话、嵌入、语音等接口的 **JSON 体字段多、嵌套深**，需要 **入参与出参校验**。FastAPI 将 **Python 类型注解** 与 **Pydantic 模型** 结合，校验错误可自动映射为 **422** 等规范响应，并生成 **OpenAPI（Swagger）** 文档，便于与 **Spring Boot 调用方** 对齐字段说明。Flask 同样可通过 **Marshmallow、Flask-Pydantic** 等扩展实现类似能力，但需 **额外选型与胶水代码**，默认形态不如 FastAPI **一体化**。

**（2）异步 IO 与外部模型调用**  
大模型、嵌入与部分语音接口属于 **高延迟、IO 密集** 调用。FastAPI 构建在 **Starlette** 之上，对 **`async`/`await`** 有良好支持，可在单进程内 **并发等待** 多个上游 HTTP 响应，提高吞吐潜力。Flask 2.x 虽具备异步视图能力，但历史项目与教材多以 **同步 WSGI** 为主；在 **以异步协程组织 LLM 调用链** 的路径上，FastAPI 的 **默认叙事与示例** 更贴近当前 **Agent 编排** 实践。

**（3）与 AI 生态的贴合度**  
LangChain、部分语音与向量相关示例与文档中，**基于 FastAPI 挂载路由** 的写法较为常见，依赖安装与 **类型提示** 风格统一，有利于 **减少样板代码**、加快迭代。

**（4）小结（可写入论文）**  
Flask 的 **「极简 + 扩展」** 在通用 Web 与教学演示中依然经典；本系统 Python 侧承担 **类型严格的 API 面 + 异步友好的模型 IO + 自动文档**，在这些维度上 **FastAPI 开箱更省工**，故作为智能服务框架。若答辩追问「是否必须用 FastAPI」，可答：**否**；在同等规范下用 Flask **亦可实现**，属于 **工程偏好与维护成本** 的取舍，而非唯一正确答案。

### 2.2.7 B/S 模式与前后端分离

本系统采用 **B/S（Browser/Server）** 模式：用户通过浏览器访问前端静态资源，业务逻辑与数据存储在服务端。前后端通过 **JSON over HTTP** 通信，有利于 **多端复用同一套 API** 与 **独立部署、独立版本演进**，与第二章技术概述及后续系统设计章节相呼应。
