## 4.1 系统架构设计

智慧食堂系统采用 **B/S 架构**：用户通过浏览器访问 **大屏展示端**、**移动 H5 端** 与 **管理后台**；服务端以 **Spring Boot** 统一提供业务数据接口与静态资源代理，以 **关系型数据库** 持久化业务实体；**智能化能力**（大模型对话、工具调用、向量嵌入、语音合成与识别等）由独立进程 **ai-service（FastAPI）** 承载，Java 后端在对话场景下承担 **编排与网关（BFF）** 职责，将前端请求转发至 AI 服务并落库会话与消息。该划分有利于 **伸缩部署** 与 **故障隔离**：业务库与对话历史由 Java 侧保证事务与一致性，模型算力与 Agent 逻辑集中在 Python 侧迭代。

### 4.1.1 前端代码分层设计

系统前端分为 **三个相互独立的工程**，技术选型统一为 **TypeScript**，构建工具为 **Vite**，通过环境变量将 API 基址指向 Spring Boot 的 `/api` 路径；智能化交互场景下，各端 **统一调用** 后端暴露的对话接口（如 `/api/ai/chat`），由后端再转发至 AI 服务，避免浏览器直连多个异构端口带来的跨域与安全配置负担。

**（1）大屏端（`screen`）**  
采用 **React**。视图以 **页面（`pages`）与公共组件（`components`）** 为主；数据获取通过 **自定义 Hook**（如 `useBackendData`）或组件内 `fetch` 访问 REST 接口；大屏 AI 助手组件负责维护会话状态、请求头中的客户端标识及 `clientType` 等字段，与后端约定一致。

**（2）移动端 H5（`mobile`）**  
采用 **React**。**`components`** 承载各业务 Tab 与详情页 UI；**`api`** 目录按领域拆分 TypeScript 模块，封装路径、方法与类型；**`lib`** 存放与界面弱耦合的纯函数逻辑（如摘要拼装）。移动端与大屏共享部分类型与客户端封装时，可依赖 **`shared/api`**，减少重复。

**（3）管理后台（`admin`）**  
采用 **Vue 3** 与 **Element Plus**。目录上体现典型 **单页应用分层**：**`pages`** 为业务页面；**`router`** 负责路由与 **导航守卫**（登录态、动态权限路由）；**`pinia`** 管理用户与权限状态；**`http`** 统一 Axios 实例与拦截器；**`common/apis`、`common/utils`、`common/constants`** 提供跨页面复用的接口封装、工具函数与常量。该结构与主流后台模板一致，便于维护与二次开发。

综上，前端分层可概括为：**视图层（页面/组件）— 路由与状态层 — 网络访问层（api/http）— 工具与常量层**；三端在「网络访问层」均与后端 REST 对齐，在智能化场景下共享同一套对话协议（消息列表、会话 id、客户端类型等）。

### 4.1.2 后端代码分层设计

后端基于 **Spring Boot**，持久化采用 **Spring Data JPA**，数据访问接口命名为 **`repository` 包**（接口继承 `JpaRepository` 等），与部分教材中 MyBatis 的 **Mapper 层** 名称不同，但职责等价，均属于 **数据访问层**。

**（1）控制层（Controller）**  
位于 `com.scs.controller` 及其子包（如 `v1`）。使用 `@RestController` 与 `@RequestMapping` 声明资源路径前缀（如 `/api/posts`、`/api/ai`）。负责 **参数解析、HTTP 状态与统一响应封装、调用下层服务或直接编排**；部分与外部 AI 服务协作紧密的接口（如 AI 对话）在控制层内完成 **会话摘要、RestTemplate 转发、结果反序列化** 等编排逻辑，属于 **薄网关 + 厚编排** 的常见工程折中。

**（2）业务层（Service）**  
位于 `com.scs.service`。将 **可复用的领域步骤** 封装为 Spring 管理的 Bean，例如向量检索、快照报表门面、商家评分聚合、管理端会话签发等。**并非每个领域对象都对应独立 Service 类**：许多 CRUD 由控制层直接调用 Repository 即可完成，论文中可如实表述为 **「以 Repository 为中心的轻量服务层，复杂场景抽取 Service」**。

**（3）数据访问层（Repository）**  
位于 `com.scs.repository`。按聚合根或表语义划分接口，由 JPA 生成查询，避免手写 SQL 分散在业务代码中。

**（4）实体层（Entity）**  
位于 `com.scs.entity`。与数据库表映射的 JPA 实体，作为 **领域模型** 在控制层、服务层与持久层之间传递。

**（5）数据传输对象（Dto）与统一响应**  
如 `ApiResult`、`TableDataDto` 等，用于 **对外 JSON 契约** 与 **管理端表格数据** 的规整，减轻实体类直接暴露带来的耦合。

**（6）配置与横切（Config）**  
如 Web MVC 配置、当前用户上下文过滤器等，支撑跨接口的用户识别与 CORS 等行为。

**关于 Constants 与 Util**  
本仓库 **未单独建立顶层 `constants`、`util` 包**；常量多以内联或各类 `private static final` 形式存在，工具型方法就近置于使用类中。论文可写：**「常量和工具方法遵循高内聚原则，随控制层或服务类定义；管理后台前端则单独划分 constants/utils 目录。」** 这样与代码一致，无需虚构目录。

### 4.1.3 RESTful API 设计

接口风格遵循 **REST 思想** 与工程可读性的平衡：**资源名使用复数英文路径**，嵌套资源通过子路径表达（如帖子评论）；**HTTP 动词** 以 `GET` 查询、`POST` 创建、`PUT`/`PATCH` 更新、`DELETE` 删除为主（具体以各控制器实现为准）。统一前缀 **`/api`** 标识业务 API；管理端认证等能力使用 **`/api/v1`** 前缀（如 `/api/v1/auth/login`），与主业务路径区分，便于版本演进。

**响应体** 广泛采用 **`ApiResult<T>` 信封结构**（业务码、提示信息、载荷数据），前端可根据 `code` 判断成功与失败，符合前后端分离协作习惯。

**典型资源示例（与实现对应，正文可择要列举）**：

| 资源前缀 | 说明 |
|----------|------|
| `/api/ai` | 智能对话、会话与消息查询 |
| `/api/vector/posts` | 帖子语义检索相关 |
| `/api/posts`、`/api/post-comments` | 动态帖子与评论 |
| `/api/vendors`、`/api/vendor-reviews` | 商家与评价 |
| `/api/orders`、`/api/menu-items`、`/api/queue-entries` | 订餐、菜品、排队 |
| `/api/lost-items`、`/api/found-items` | 失物与招领 |
| `/api/v1/data`、`/api/v1/auth` | 管理端数据入口与登录 |

智能化扩展能力由 **ai-service** 提供额外路径（如 `/api/chat`、`/embeddings`、语音相关路径等），由 Spring Boot **服务端转发** 调用，浏览器不直接持有 AI 服务密钥，安全性更好。

### 4.1.4 RBAC 模型设计

本系统在 **角色访问控制（RBAC）** 上采用 **轻量级、可落地的实现**，与「用户—角色—权限」经典模型对应关系如下。

**（1）用户与角色**  
用户实体包含 **角色字段**（如学生、管理员、商家等，以实际库中取值为准）。角色决定 **能否登录管理后台**：认证接口校验用户名密码后，仅允许 **管理员、商家** 等具备后台准入的角色进入，否则返回 **禁止访问** 的业务码。

**（2）管理后台前端权限**  
Vue 管理端在 **路由守卫** 中根据 token 拉取用户信息，得到 **角色数组**，由权限模块 **动态注册可访问路由**，实现 **菜单与页面的可见性控制**，属于 **表现层 RBAC**。

**（3）与 Agent 工具权限的衔接**  
AI 对话请求中，Java 后端将用户标识与 **角色** 一并传入 Python 侧；AI 服务内根据 **客户端类型（admin / screen / mobile）与角色** 配置 **工具白名单**，限制可调用的业务工具范围（例如移动端数据域闭包、访客无工具等）。这样 **RBAC 从「人能进哪些页面」延伸到「智能体能执行哪些动作」**，与多模态、Agent 选题自然衔接。

若论文需对比标准 RBAC，可说明：当前 **未引入独立权限表与 Spring Security 方法级注解全覆盖**，接口级控制依赖 **登录会话、业务校验与 AI 侧工具策略**；后续可扩展为 **权限字 + 接口注解** 的细粒度模型。

### 4.2.4 向量检索与帖子智能模块

**设计意图与功能概述**：本模块用于支持 **语义搜索** 与 **知识召回**。传统关键词检索在面对口语化、同义改写较多的自然语言表述时，容易出现 **漏召回**；向量检索通过将文本映射到向量空间并按相似度排序，能够缓解「说法不同、意思相近」带来的匹配困难，因而适合作为帖子、公告等 **非结构化文本** 的辅助检索手段。

在本系统中，帖子标题与正文可先经 **向量化** 写入 **`post_vector_index` 向量索引**；检索时由服务端对查询文本使用同一套嵌入流程生成查询向量，与索引中的向量计算相似度，取 Top-K 后 **回表** 返回帖子业务字段。在 **智能问答 / 检索增强（RAG）** 语境下，上述召回结果还可作为 **证据上下文** 供大模型组织回答，使输出更贴近系统内真实帖文（具体检索与嵌入提供方、降级策略及与 Agent 的衔接边界见 **`docs/AI板块与RAG技术设计说明.md`**、**`docs/后端设计（Spring Boot）.md` §2**）。

模块侧重点不在于堆砌复杂检索算法，而在于：**让用户用更自然的表述也能找到相关信息**，并为后续扩展（分块、过滤、重排、与对话工具链打通）预留接口。

**分层设计**：  
- **V**：**移动端**「本周食堂圈 · AI 小结」等场景在配置后端地址时，可调用向量检索结果做 **检索增强展示**；常规动态/反馈列表仍以 `PostController` 等列表接口为主。大屏与后台管理端可按产品规划增加独立搜索入口，与本文 **C** 层 API 对接。  
- **C**：`PostVectorSearchController`（`/api/vector/posts/search`、`/rebuild`）、以及帖子主数据 `PostController` 等。  
- **M**：`PostVectorIndex` 与 `PostVectorIndexRepository`；**`PostVectorSearchService`** 负责嵌入请求路由（含 **ai-service** `/api/embeddings`、可选 OpenAI、本地降级）、相似度计算与结果组装。

**配图（建议 图 4-9）**：建议展示 **「搜索/查询输入 →（可选）嵌入服务 → 向量相似度检索 → 帖子向量索引 + 业务帖子表回表 → 结果返回」** 的数据流；嵌入服务在图中可与 **Spring Boot** 并列引出 **ai-service（虚线）**，以体现 **索引构建与查询向量化** 的来源。

### 4.2.9 管理端统一数据与审计模块

**功能概述**：管理端通过 **v1 数据接口** 维护多表数据；**审计日志** 记录关键操作；AI 服务可在对话结束后 **异步写审计**，支撑可追溯。

**分层设计**：  
- **V**：admin 表格页、CRUD 表单。  
- **C**：`V1DataController`、`TablesController`、`AuditLogController` 等。  
- **M**：`AuditLog` 及 Repository；各业务表实体。

**配图（建议 图 4-14）**：**V** 表格 → **C** V1Data → **M** 多表 + **audit_logs**。

### 4.2.10 扩展代理请求与其它接口

**功能概述**：`AgentRequestController` 等用于与智能体相关的请求登记或扩展；`SensorLog`、`CallEvent` 等物联网或呼叫类数据接口可按实际论文范围取舍描述。

**分层设计**：按具体控制器与实体 **套同一 MVC 模板** 简述即可，避免章节过长。
