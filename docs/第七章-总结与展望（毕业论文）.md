## 7.1 工作总结

本文围绕校园 **智慧食堂** 场景，设计并实现了一套 **多终端、前后端分离** 的信息化系统。在总体架构上，系统采用 **B/S 模式**：**大屏展示端**、**移动 H5 端** 与 **管理后台** 分别基于 **React** 与 **Vue 3** 等现代前端技术构建，通过统一的 **RESTful API** 访问 **Spring Boot** 业务服务；核心业务数据持久化于 **MySQL**，接口契约与统一响应结构便于多端复用与联调。

在 **业务功能** 层面，系统覆盖了食堂运营中的典型需求，包括 **用户与认证**、**商家与菜品浏览**、**订单与排队**、**动态社区与互动**、**失物招领**、**食安与库存相关数据管理** 等，并在管理端提供 **统一数据入口** 与 **审计留痕** 能力，支撑日常运维与追溯。

在 **智能化** 层面，系统将大模型应用从「单次问答」提升为 **可编排、可约束** 的工程形态：**Spring Boot** 承担会话持久化、上下文摘要与对 **FastAPI ai-service** 的转发编排；**LangChain / LangGraph** 在 Python 侧实现 **工具调用闭环**，将订单、帖子、商家等业务能力封装为 **HTTP 工具**，并结合 **客户端类型与角色** 实施 **工具白名单与移动端数据域约束**，使智能助手在访问业务数据时具备可解释的权限边界。**审计日志** 以 best-effort 方式记录对话与工具调用摘要，为「做了什么」提供最小可追溯依据。此外，系统还集成了 **语音合成与识别** 等能力，支撑大屏等场景下的语音交互链路（具体边界以语音专项设计说明为准）。

在 **检索增强** 方向，本文在帖子域实现了 **向量索引构建与 Top-K 相似度检索** 管线：**帖子正文经向量化写入独立索引表**，查询时由服务端完成嵌入与排序并回表返回业务字段；**移动端「本周食堂圈 · AI 小结」** 已将上述检索结果以 **检索增强展示** 的形式并入周报详情，使用户在自然语言语境下获得与本周讨论相关的 **关联帖证据**。嵌入提供方支持 **本地 AI 服务、OpenAPI 类接口与降级策略**，便于演示环境与后续替换真实嵌入模型。

综上，本文完成了从 **需求分析、系统设计、模块实现到测试方案** 的完整工程闭环，形成了一套 **业务可靠、智能可扩展、结构可答辩** 的智慧食堂原型系统。

## 7.2 主要特点与收获

结合实现过程，本工作的特点可归纳为以下几点。

（1）**职责分离清晰**：事务型业务与会话状态由 **Java** 侧保障一致性，模型推理与 Agent 编排集中在 **Python** 侧迭代，有利于团队分工、独立扩缩容与故障隔离。

（2）**智能体工程化**：通过 **Tool 配置化、LangGraph 状态机、分端系统提示词** 等手段，将「模型能说什么」与「系统允许做什么」分离；**Guardrails** 主要体现在 **工具面与数据域** 的工程约束上，而非仅依赖提示词自觉。

（3）**检索与业务解耦**：帖子向量索引与主表分离，检索服务可替换嵌入模型并 **全量重建**；检索结果既可服务未来的 **生成式 RAG**，也已接入 **移动端周报的证据展示**，体现「检索层先行、产品形态渐进」的落地路径。

（4）**文档与代码可追溯**：需求章、设计章、专项设计说明（AI、RAG、语音、后端等）与测试章相互对照，便于答辩时快速定位 **设计决策与代码路径**。

通过本课题，作者在 **全栈协作、REST 契约设计、多服务集成与 AI 应用治理** 等方面得到了系统训练，对「智能系统不仅是模型调用，更是约束与运维」有了具体认识。

## 7.3 存在的问题与局限

如实说明局限有利于体现工程成熟度判断，也与各专项文档中的 **实现边界** 保持一致。当前系统仍存在以下不足。

（1）**向量与语义检索**：默认链路下，嵌入实现包含 **可复现哈希占位向量**，与商业 **神经语义嵌入** 在效果上不可等同；索引采用 **MySQL 存储 + 应用层全量扫描相似度**，数据量增大时 **延迟与扩展性** 将成为瓶颈；**元数据过滤、分块、重排** 等工程增强尚未实现。

（2）**RAG 闭环程度**：检索结果已在 **移动端周报** 中展示，但 **统一的服务端「检索—拼 prompt—生成」接口** 与 **对话 Agent 侧的向量检索工具** 尚未纳入主线实现，智能问答场景下模型仍主要依赖 **列表类工具** 访问帖文，与「完整 RAG 问答」尚有距离。

（3）**智能化纵深**：**内容型安全护栏**（注入检测、敏感回复策略等）未单独模块化；**多智能体协作**、**自动化离线评测**（如 Recall@K、忠实度）未开展；**语音与流式传输** 以当前设计说明中的实现边界为准，不宜表述为生产级实时系统。

（4）**测试与运维**：测试以 **黑盒手工与轻量性能观察** 为主，**接口自动化回归、压力测试与安全渗透** 未作为本文重点；**快照报表、周期 AI 报告** 等能力在数据层与接口上已预留，与订单发帖的 **全自动聚合生成** 衔接程度需以库内实际数据与任务为准，避免过度承诺。

## 7.4 后续工作展望

基于上述总结与局限，未来可从以下方向延续与深化。

（1）**检索基础设施升级**：引入 **专用向量库或带向量扩展的数据库**，采用 **ANN 索引** 降低检索延迟；将长文本 **分块** 后再向量化，并支持 **时间、档口、反馈类型** 等 **元数据过滤**；可选接入 **重排模型** 提升 Top-K 质量。

（2）**嵌入与 RAG 产品化**：替换为 **开源或商用神经嵌入模型**，规范 **版本与重建流程**；实现 **服务端 RAG 回答接口** 或将 **`/api/vector/posts/search` 注册为 Agent 工具**，使对话与检索共用同一套证据链；建立小规模 **评测集** 支撑对比实验与论文数据。

（3）**治理与体验**：加强 **审计字段与可视化**；探索 **内容型护栏** 与 **拒答策略**；在管理端或大屏增加 **语义搜索入口**（与现有 API 对接），统一各端检索体验。

（4）**工程质量与部署**：补充 **自动化接口测试与 CI**；在 **校园网或容器化环境** 试点部署，观察 **并发、日志与备份** 需求；对 **商家结算、支付对接、消息推送** 等真实运营能力按业务优先级分阶段扩展。

通过以上工作，可在现有架构上逐步从 **毕设级原型** 演进为 **更可运维、更可评测** 的智慧食堂信息系统，使智能化能力真正服务于日常管理与师生体验。
