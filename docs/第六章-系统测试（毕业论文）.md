# 第六章 系统测试

> **使用说明**：本文稿与「智慧食堂 SCS」多终端 + Spring Boot + 独立 AI 服务（FastAPI）的实现相对应。复制到 Word 后请按学院模板修改 **章号、节号、表号、图号**；表中「实际结果」「是否通过」请在自行执行测试后据实填写（未测项勿填「通过」）。

---

## 6.1 测试意义

软件测试是保障系统质量的重要手段。智慧食堂系统在业务上覆盖 **大屏展示、移动 H5、管理后台** 三类终端，在技术上涉及 **前后端分离、REST 接口、会话与权限控制** 以及 **智能对话与外部 AI 服务编排** 等链路，模块间依赖关系较多。若在集成阶段缺乏系统性验证，容易出现接口契约不一致、跨端状态不同步、权限边界遗漏或异常场景处理不当等问题，影响可用性与可维护性。

通过制定测试方案并设计用例，可以在 **功能层面** 验证系统是否满足需求与设计说明中的行为描述，在 **非功能层面** 对典型场景下的响应与资源占用形成客观记录，为后续迭代与答辩演示提供依据。对本毕设而言，测试还具有 **工程完整性** 意义：表明实现不仅在结构上分层清晰，且在可观察行为上与预期一致。

---

## 6.2 测试目的

本阶段测试的主要目的如下。

（1）**功能正确性**：验证用户认证、业务 CRUD、多终端页面与接口联动、AI 对话及会话持久化、管理端统一数据入口与审计等核心流程能够按设计完成，返回数据与界面展示符合预期。

（2）**异常与边界行为**：验证非法输入、未登录访问受保护资源、无权限角色访问管理端等情况下，系统能否返回统一错误结构或友好提示，而不出现未处理异常暴露栈信息等不当行为。

（3）**集成与兼容性**：验证浏览器端访问各工程构建产物时，与 Spring Boot `/api` 及 AI 相关转发链路能够正常协作；在开发与联调所用环境下端到端路径可重复执行。

（4）**性能与体验（轻量）**：在单机或局域网部署条件下，对主要列表查询、首屏加载及 AI 首轮响应等进行 **观察性** 记录，判断是否存在明显超时或阻塞，为毕设演示场景提供参考（本文不宣称完成生产级压力测试）。

## 6.3 测试方案

### 6.3.1 测试方法

采用 **黑盒测试** 为主：依据需求与设计说明，从用户与管理员视角设计输入与操作步骤，检查输出、界面状态与接口返回，不关心内部实现细节。功能测试以 **手工执行** 为主；性能相关观察可结合浏览器 **开发者工具（Network）** 查看关键请求耗时。

### 6.3.2 测试环境

测试环境与开发、联调环境一致或为其子集，典型配置可归纳如表 6-1（可按实际机器修改）。

**表 6-1 测试环境说明**

| 项目 | 说明 |
|------|------|
| 操作系统 | Windows 10/11 或 Linux（与部署一致） |
| 运行环境 | JDK 17（或与项目 `pom.xml` 一致）、Node.js（前端构建）、Python（ai-service，若启用） |
| 数据库 | MySQL，使用测试库或本地库；可配合初始化脚本导入最小样本数据 |
| 服务端 | Spring Boot 业务服务；按需启动 FastAPI **ai-service** |
| 客户端 | Chrome / Edge 等现代浏览器；移动端 H5 可用浏览器设备模拟或实机 |
| 网络 | 本机或局域网，API 基址通过环境变量指向后端 `/api` |

### 6.3.3 测试数据与范围

**不要求** 使用大规模真实业务数据。测试数据采用 **手工构造的最小集合** 即可，例如：具备后台准入角色的管理员账号与普通业务用户各至少一个、若干菜品与商家记录、少量订单或帖子样本等。数据仅需支撑用例中的前置条件与操作步骤。

测试范围覆盖第四章所述主要模块：**认证与用户、智能对话与 Agent、帖子与向量检索、商家菜品与订单排队、动态社区、失物招领、食安与库存相关接口（若前端已接入）、管理端 v1 数据与审计** 等；对未实现或未接界面模块可在表中标注「不适用」或删减对应用例。

### 6.3.4 通过准则

单条用例 **实际结果与预期结果一致** 且无阻碍性缺陷，则判定为通过。若存在缺陷，修复后应对受影响用例执行 **回归测试**。本章所列用例全部通过后，可认为本阶段功能测试达到毕设验收与演示要求（在所述环境范围内）。

### 6.4.2 性能测试

本毕设 **未按生产标准** 开展独立压力测试工具（如 JMeter）的长期压测与容量规划。性能验证采用 **轻量、可重复** 的方式，目标是在既定环境下确认 **无明显性能瓶颈阻碍演示与常规操作**。

**（1）测试指标与方式**

- **页面与接口耗时**：使用浏览器开发者工具 **Network** 面板，记录主要页面 **DOMContentLoaded / Load** 及关键 XHR 的 **Waiting（TTFB）** 与总耗时，每个场景重复 3～5 次取 **主观可接受** 结论即可。  
- **AI 首轮响应**：从发送消息到首包或可读回复出现的时间，受模型与网络影响较大，记录 **数量级**（如数秒内完成）即可，不必与公网 SaaS 对标。  
- **并发**：若有条件，可两人同时操作管理端与移动端做 **简单并发** 观察；未做严格并发实验则如实写明。

## 6.5 本章小结

本章说明了智慧食堂系统测试的意义与目的，给出了以黑盒为主、手工执行并结合浏览器工具进行轻量性能观察的 **测试方案**，明确了 **测试环境与最小样本数据** 的组织方式及通过准则。在功能测试部分，按 **认证与用户、智能对话与 Agent、帖子与向量检索、商家订单与评价、失物招领、食安库存、管理端数据与审计及统一接口** 等模块设计了多组用例，便于逐项验证系统行为与设计的一致性。在性能部分，采用 **非压测** 的观察性记录，避免缺乏工具与数据时夸大测试深度。后续若在真实校园环境部署，可在本章方法基础上扩展 **自动化回归、接口契约测试与正式压力测试**，以进一步提升质量保障水平。
