## 3.1 可行性分析

### 3.1.1 经济可行性

本系统定位为 **毕业设计与校园信息化演示** 场景，建设路径以 **开源框架与自建部署** 为主。前端采用 **Vite + TypeScript** 与 **React / Vue 3** 生态，后端采用 **Spring Boot**、数据库采用 **MySQL**，智能化侧采用 **FastAPI** 与 **LangChain / LangGraph** 等 **可本地或按量调用** 的模型接口；除 **服务器/云主机、域名与可选的大模型 API 资费** 外，无需采购商业中间件许可证即可完成开发与联调。相较一次性采购大型商业餐饮 SaaS 或闭源智能中台，本方案 **初期投入可控**，适合教学与原型验证阶段，后续若在校园内推广，可在现有架构上 **横向扩容进程与数据库** 再评估专项预算。

### 3.1.2 技术可行性

系统采用 **B/S 架构** 与 **前后端分离**，技术栈均为业界常见选型：**Spring MVC + Spring Data JPA** 支撑事务型业务与 REST 契约；**独立 Python 智能服务** 承载大模型编排、嵌入与语音，与 Java 通过 **HTTP** 集成，职责清晰、便于分工开发与故障隔离。向量检索采用 **关系库内索引表 + 服务端相似度编排**，避免毕设阶段引入过多异构存储组件。相关技术文档与社区资源丰富，**开发与调试工具链成熟**，在指导教师与课程训练范围内 **具备可实现性**。主要风险在于 **外部模型服务稳定性与密钥管理**，可通过 **环境变量配置、服务端转发、降级策略（如无工具时纯文本回复）** 缓解，不构成颠覆性障碍。

### 3.1.3 操作可行性

**终端侧**：师生与管理员通过 **浏览器** 访问大屏页、移动 H5 与管理后台，**无需安装专用客户端**（移动端亦可添加到主屏幕），学习成本较低。**管理端** 采用 **Element Plus** 等成熟组件库的表格、表单与导航，符合常见后台操作习惯。**业务侧**：订餐、浏览动态、失物招领等流程与日常 App 使用方式接近；AI 助手以 **对话窗口** 呈现，降低非技术用户使用门槛。系统若部署在校园网或内网，可通过 **统一说明文档与简短培训** 完成上手，**操作可行性良好**。

### 3.2.1 系统角色

系统参与方按 **使用端与职责** 划分，与轻量级 **RBAC** 设计一致：用户实体带有 **角色** 信息，用于 **能否进入管理后台**、**管理端菜单可见性** 以及 **AI 侧工具白名单与数据域** 等策略。典型角色可归纳如下（具体枚举以数据库与实现为准）。

**表 3-1 角色与行为概述（示例）**

| 角色 | 主要使用端 | 核心诉求 |
|------|------------|----------|
| 师生用户（含在校学生等） | 大屏、移动 H5 | 浏览资讯与菜品、下单/排队、社区互动、失物招领、使用 AI 问答（受工具与数据域约束） |
| 访客 / 未登录用户 | 大屏、移动 H5（部分功能） | 浏览公开内容；敏感写操作与个性化能力受限；AI 能力可按设计降级（如无工具） |
| 食堂管理员 | 管理后台 | 用户与业务数据维护、食安与库存相关数据管理、审计查询等 |
| 商家（若启用） | 管理后台 | 与本档口相关的数据维护；登录受认证策略约束 |
| 系统运维 / 开发人员 | 部署环境 | 配置环境变量、启停服务、观察日志（非功能性角色，论文可简写） |

**需求要点**：管理后台 **仅允许具备准入角色** 的用户登录；各端访问 AI 时须携带 **客户端类型（admin / screen / mobile）**、**客户端标识** 及 **会话 id** 等约定字段，以便后端与 **ai-service** 实施 **分端提示词与工具策略**（与 **4.2.1、4.2.3** 设计一致）。

### 3.2.2 多终端接入与展示需求（对应第四章 **4.2.1**）

**需求描述**：系统应为大屏展示端（`screen`）、移动 H5 端（`mobile`）与管理后台（`admin`）分别提供 **入口、布局与导航**；三端均能 **统一配置后端 API 基址**，通过 REST 访问同一业务域。在智能化场景下，各端向服务端提交对话等请求时，须携带 **`clientType`、客户端 id、会话 id** 等与后端及 **ai-service** 约定一致的参数，以支撑 **分端展示与权限策略**。

**验收要点**：三端可独立构建与访问；跨域与静态资源访问符合部署配置；AI 相关请求字段与第四章 **4.2.1** 所述一致、可被后端正确解析并转发。

---

### 3.2.3 用户与认证需求（对应第四章 **4.2.2**；含登录/注册）

**需求描述**：**（1）业务侧用户**：通过用户资源相关接口完成 **信息查询与更新**（以实际接口为准）；涉及身份的操作须与登录态或 Token 机制一致。**（2）管理后台**：提供 **账号密码登录**；支持 **验证码**（可配置关闭）；登录成功后建立 **会话或 Token**，结合用户表完成校验；**不具备后台准入角色** 的用户须被拒绝并返回约定业务码。**（3）登录态与路由**：管理端通过 **路由守卫** 控制未登录访问；退出登录后清除凭证。**（4）注册**：若实现自助注册，须校验必填项、唯一性与格式；若以管理端开户为主，需求可表述为 **管理员维护用户主数据**。

**验收要点**：与 **4.2.2** 中 `UserController`、`AuthController`（`/api/v1`）、`V1UserController` 等职责一致；未登录访问受保护资源时返回 **统一错误结构或 401**。

---

### 3.2.4 智能对话与 Agent 编排需求（对应第四章 **4.2.3**）

**需求描述**：系统应支持 **多轮智能对话**；**会话与消息** 须 **持久化** 至业务库；对长对话应支持 **摘要压缩** 以控制上下文长度。用户消息经 **Spring Boot** 转发至独立 **ai-service**，由 **LangChain / LangGraph Agent** 执行推理与 **工具调用**；返回结果须能解析 **工具调用轨迹、建议问句** 及 **结构化内容**（如推荐菜品卡片），并与文本一并呈现给各端前端。须满足 **权限与降级**：按角色与客户端类型限制工具面，无权限时 **不越权执行业务写操作**，可按设计 **降级为纯文本或提示**。

**验收要点**：对话可连续多轮；刷新或重进可加载历史；与 **4.2.3** 中 `AiChatController`（`/api/ai`）及会话实体设计一致；浏览器不直连模型密钥端口。

---

### 3.2.5 向量检索与帖子智能需求（对应第四章 **4.2.4**）

**需求描述**：帖子正文应支持 **向量化** 并与检索能力结合，使用户可通过 **语义搜索** 发现相关动态（与 RAG 设计文档一致处可在正文互引）。移动端与大屏应提供 **检索入口**；服务端应完成 **嵌入请求调用**（如调用 ai-service）、**索引维护** 与 **相似度排序**，并将结果与帖子主数据 **回表组装** 后返回。检索失败或索引未就绪时应有 **可理解提示**，而非未处理异常。

**验收要点**：与 **4.2.4** 中 `PostVectorSearchController`（`/api/vector/posts`）、`PostVectorSearchService`、`PostVectorIndex` 等设计一致；结果可追溯到具体帖子。

---

### 3.2.6 动态社区与互动需求（对应第四章 **4.2.5**）

**需求描述**：用户应能 **浏览、发布帖子**；在详情页进行 **评论**；支持 **点赞** 等互动（以实现为准）。发布与评论内容须通过 **基本校验**（非空、长度等）；操作与 **登录态** 绑定关系符合业务规则。

**验收要点**：与 **4.2.5** 中 `PostController`、`PostCommentController` 及 `Post`、`PostComment`、`PostLike` 等模型一致。

---

### 3.2.7 商家、菜品与评价需求（对应第四章 **4.2.6**）

**需求描述**：师生应能 **浏览档口（商家）与菜品** 信息；完成 **订餐** 与 **排队** 相关操作（流程以实际页面为准）；对商家提交 **评价** 后，系统应维护评价数据并支持 **评分聚合** 展示或查询，与 **VendorRatingAggregateService** 一类设计目标一致。

**验收要点**：与 **4.2.6** 中 `VendorController`、`MenuItemController`、`VendorReviewController`、`OrderController`、`QueueEntryController` 等划分一致。

---

### 3.2.8 失物招领需求（对应第四章 **4.2.7**）

**需求描述**：用户应能发布与浏览 **失物**、**招领** 信息；支持 **评论** 交互（若前端已接入）。列表与详情字段完整，创建与评论权限符合业务约定。

**验收要点**：与 **4.2.7** 中 `LostItemController`、`FoundItemController`、`LostItemCommentController` 及对应实体一致。

---

### 3.2.9 食安、食材与库存需求（对应第四章 **4.2.8**）

**需求描述**：系统应支撑食堂 **安全管理** 相关数据能力：**食材**、**过敏原公示**、**检验报告**、**留样**、**库存流水** 等的 **维护与查询**（展示端可为管理后台或大屏，以实际前端为准）。数据录入须满足 **基本完整性与关联关系**（如食材与菜品关联，以库表为准）。

**验收要点**：与 **4.2.8** 中 `MaterialController`、`AllergenDisclosureController`、`TestReportController`、`RetainedSampleController`、`StockMovementController` 等一致；未做前端接入的子项可在论文中标 **「后台接口就绪，界面可选」**。

---

### 3.2.10 管理端统一数据与审计需求（对应第四章 **4.2.9**）

**需求描述**：管理端应能通过 **v1 数据接口** 等对 **多表业务数据** 进行维护（表格、表单 CRUD）；**审计日志** 应记录关键操作（操作者、动作、对象等），支持查询与追溯。**AI 服务** 在对话结束后宜支持 **异步写入审计**（best-effort），与 **可追溯** 设计一致。

**验收要点**：与 **4.2.9** 中 `V1DataController`、`TablesController`、`AuditLogController` 及 `AuditLog` 实体一致；敏感写操作可对应审计记录（以实现为准）。

### 3.2.11 扩展代理与其它接口需求（对应第四章 **4.2.10**）

**需求描述**：若论文纳入 **智能体相关请求登记或扩展**（如 `AgentRequestController`），应明确 **登记字段、查询方式** 与 **权限**。**传感器日志、呼叫事件** 等物联网或呼叫类数据接口，按毕设范围 **取舍描述**：纳入则给出 **采集、存储与查询** 需求；不纳入则本章删节并在第四章同步删减，保持 **需求—设计一致**。

**验收要点**：仅对正文宣称已实现的接口提验收项，避免需求悬空。

## 3.3 系统性能需求分析

本系统以 **校园食堂场景下的演示与日常使用** 为目标，**不套用互联网峰值电商级指标**，性能需求表述为 **可验证、可观察** 的区间即可。

**（1）响应时间**  
在 **局域网或校园网** 典型环境下，常规 **列表查询与详情接口** 应在 **秒级内** 返回可读结果；**AI 首轮回复** 受模型与链路影响较大，需求可表述为 **在可接受等待时间内返回可读内容**，并允许配置超时与重试策略。

**（2）并发与容量**  
支持 **多名用户同时** 浏览与提交业务请求；管理端 **少量并发** 维护操作不导致数据错乱。毕设阶段以 **功能正确性优先**；若未做正式压测，论文可写明 **「后续可结合压测工具制定容量基线」**。

**（3）可用性与降级**  
当 **ai-service** 或外部模型不可用时，**核心业务（浏览、下单、后台维护）** 仍应 **尽可能可用**；AI 能力可 **提示服务不可用或降级为无工具对话**，避免拖垮主业务线程（具体策略以实现为准）。

**（4）资源占用**  
在开发与演示所用 **单机或低配虚拟机** 上可完整运行 **Java 后端、MySQL、（可选）Python 服务**，便于答辩环境复现。

## 3.4 接口需求

**（1）风格与契约**  
对外采用 **RESTful 风格** 的 **JSON API**：资源路径清晰（如 `/api/posts`、`/api/ai/...`）；管理端认证与部分管理能力使用 **`/api/v1`** 前缀（如 `/api/v1/auth/login`），便于版本演进。

**（2）统一响应**  
业务接口广泛使用 **`ApiResult` 一类信封**：包含 **业务码、提示信息、数据载荷**，便于前端统一处理成功与失败。

**（3）安全与跨域**  
浏览器端仅访问 **Spring Boot 暴露的源**；**CORS** 与 **凭证（Cookie/Token）** 策略与前端工程一致。敏感能力 **不得在浏览器暴露第三方密钥**。

**（4）与智能服务的内部接口**  
Java 与 **ai-service** 之间通过 **HTTP** 交换对话、嵌入、语音等请求；字段名与 **clientType、userId、role** 等上下文须 **与权限与审计设计一致**。

**（5）错误处理**  
参数校验失败、未认证、无权限、资源不存在等场景应返回 **可区分的业务码或 HTTP 状态**，响应体 **可读、可记录**，避免直接返回堆栈给客户端。

## 3.5 本章小结

本章从 **经济、技术、操作** 三方面论证了系统建设的 **可行性**；在 **3.2 系统功能需求分析** 中，先给出跨模块的 **角色与 RBAC 要点**，再按 **第四章 4.2.1～4.2.10** 的模块划分 **逐项** 写了 **多终端接入与展示、用户与认证、智能对话与 Agent、向量检索与帖子、动态社区、商家菜品与评价、失物招领、食安与库存、管理端数据与审计、扩展与其它接口** 的需求与验收要点，形成与 **系统设计章模块节** 的 **一一对照**，便于后续评审与追溯。另对 **性能**（响应、并发、降级、部署资源）与 **REST 接口**（资源前缀、统一响应、安全、与 ai-service 协作、错误处理）作了约束性说明。具体字段与路径以 **数据库设计与实现代码** 为准，可在迭代中细化而不改变本章与第四章的模块对应关系。
