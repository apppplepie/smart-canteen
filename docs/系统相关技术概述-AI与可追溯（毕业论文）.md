## 2.3 智能服务运行时与 Python 依赖栈

独立智能服务 **`ai-service`** 以 **FastAPI** 暴露 HTTP 接口（详见前文 **2.2.5、2.2.6**），进程启动依赖 **Uvicorn**（ASGI 服务器）。依赖栈与课题相关的条目可概括为下表（版本以 `requirements.txt` 为准，正文可写「不低于所列主版本」）。

| 依赖 | 作用（论文表述） |
|------|------------------|
| **FastAPI** | Web 框架：路由、依赖注入、OpenAPI 文档 |
| **Uvicorn** | ASGI 服务进程，承载异步请求 |
| **Pydantic v2** | 请求体验证、配置模型与序列化 |
| **httpx** | 调用大模型与上游 HTTP（较 `requests` 更利于异步与超时控制） |
| **python-dotenv** | 本地与部署环境变量管理 |
| **langchain-openai**、**langchain-core** | 与大模型供应商 API 对接及通用抽象（消息、工具协议等） |
| **langgraph** | 有状态 **Agent 图编排**（推理—工具—再推理循环） |

**为何保留 `requests`**：历史代码或与部分同步调用路径并存时可作为补充；正文可简写为「以 httpx 为主、requests 为辅」或按仓库实际删减情况表述。

**选型小结**：该栈将 **「Web 边界」**（FastAPI）、**「模型协议」**（LangChain 生态）与 **「流程控制」**（LangGraph）拆开选型，避免自研一套脆弱的 JSON 胶水层。

## 2.4 大模型应用框架：LangChain、LangGraph 与同类方案对比

### 2.4.1 本系统采用的技术组合

- **LangChain（含 langchain-core / langchain-openai）**：统一 **消息结构、工具（Tool）描述、与聊天模型的调用协议**，便于把「业务 HTTP 能力」封装成模型可调用的 **结构化动作**。  
- **LangGraph**：在 Tool 调用之上构建 **显式状态图**（Agent 节点与 Tool 节点、条件边），实现 **多步推理闭环**，而非单次 `chat.completions` 即返回终稿。

### 2.4.2 与「直接调用官方 SDK」的对比

| 维度 | 仅使用 OpenAI（等）官方 HTTP/SDK | 本系统 LangChain + LangGraph |
|------|-----------------------------------|-------------------------------|
| 工具调用 | 需自行解析 `tool_calls`、拼装多轮消息 | 框架层对 Tool 协议与重试边界有成熟抽象 |
| 流程控制 | 手写 while 循环，易散落各处 | 图编排，推理路径相对 **可叙述、可对照代码** |
| 可维护性 | 演示级代码快；工程级易膨胀 | 模块边界清晰，便于答辩时指到 `ai_agent.py` |

**选用理由**：课题强调 **Agent 与业务系统协作**，需要 **可重复的工具调用循环** 与 **可裁剪的工具集合**；直接 SDK 足以做 Demo，但 LangGraph 将 **「Agent↔Tools」** 提升为 **显式图结构**，更契合论文对 **方法可描述性** 的要求。

### 2.4.3 与 LlamaIndex、Haystack 等「以检索为中心」框架的对比

**LlamaIndex**、**Haystack** 等框架强项在 **文档索引、查询引擎、RAG Pipeline** 的一体化抽象。本系统 **检索与索引构建** 主要落在 **Spring Boot**（帖子业务表 + `post_vector_index`、相似度排序服务），**生成与工具编排** 落在 **ai-service**；RAG 呈现 **「检索在 Java、推理与工具在 Python」** 的 **双服务拆分**，而非单一 Python RAG 框架包打天下。

**选用理由**：避免在 Python 侧再维护一套与 **MySQL 业务库** 强重复的索引 ORM；Java 侧事务与 CRUD 已为权威，向量检索作为 **增强能力** 挂载其上，论文中可强调 **职责分离** 与 **可替换嵌入提供方**（见 RAG 设计说明）。

### 2.4.4 与多智能体框架（如 AutoGen、CrewAI）的简要对比

**AutoGen、CrewAI** 等强调 **多角色智能体协作、任务委派**。本系统当前以 **单用户会话 + 单 Agent 图 + 多工具** 为主，治理上采用 **客户端类型（admin/screen/mobile）与角色白名单** 裁剪工具面，属于 **「单图多工具」** 路径。

**选用理由**：食堂场景下 **首要矛盾** 是 **业务数据可信访问与权限**，而非多 Bot 对话；LangGraph 已满足 **工具循环**；多智能体可作为 **展望** 写入结论，避免正文过度承诺。

## 2.5 检索增强（RAG）与向量相关技术

### 2.5.1 RAG 范式简述

**RAG（Retrieval-Augmented Generation）** 将 **非参数化检索结果** 作为上下文，再交由大模型生成回答，以缓解 **幻觉** 并提高与 **业务语料** 的一致性。经典工作可参考 Lewis 等（2020）对知识密集任务的论述（具体文献按学校格式著录）。

### 2.5.2 与纯关键词检索的对比

| 方式 | 优点 | 局限 |
|------|------|------|
| SQL `LIKE` / 分词检索 | 实现简单、可解释 | 同义表达、口语化描述 **漏召回** |
| 向量语义检索 | 捕捉语义近邻 | 依赖嵌入模型与索引维护成本 |

**选用理由**：食堂反馈、动态帖子存在 **表述多样**，引入 **向量索引**（`post_vector_index`）提升召回；关键词检索仍可作为 **精筛或辅助**（按实现情况表述）。

### 2.5.3 与专用向量数据库的对比

业界常采用 **Milvus、Qdrant、pgvector** 等向量引擎。本系统将向量 **落 MySQL** 与业务 **帖子主表分离**，工程上 **复用现有运维栈**，适合毕设 **快速落地与演示**；正文可诚实写 **「规模扩大时可迁移至专用向量库」** 作为演进方向。

### 2.5.4 本系统分工（与双服务架构一致）

- **嵌入（Embedding）**：由 **ai-service** 提供 HTTP 能力（如 `/api/embeddings`），Spring Boot 在索引构建与检索时调用。  
- **索引与检索编排**：**Spring Boot** `PostVectorSearchService` 等负责 **重建、相似度计算、回表组装结果**。  

该划分与 **2.2.5** 中「Java 守业务、Python 守模型侧能力」的叙述一致，便于全文互引。

## 2.6 语音与多模态相关技术

### 2.6.1 能力边界

本系统语音链路按 **STT（语音转文字）** 与 **TTS（文本转语音）** 拆分实现，避免将 **识别与合成** 混为同一框架（如勿将 Coqui 误写为同时承担 ASR）。

### 2.6.2 技术选型与对比

| 能力 | 主路径 | 回退路径 | 对比说明 |
|------|--------|----------|----------|
| **STT** | 本地 **Whisper** | 云端 ASR（如豆包） | 本地降低外网依赖；云端保障可用性 |
| **TTS** | 本地 **Coqui** | 云端 TTS（如豆包） | 同上 |

**选用理由**：大屏闭环 **STT → Agent → TTS** 需要 **低时延与可离线演示**；双路径避免单点故障。涉及 **声纹与语音克隆合规** 时按学院要求单独撰写伦理与数据来源说明（参见语音专项设计文档中的提示）。

### 2.6.3 与「纯云端语音 API」的对比

纯云端方案（单一厂商 ASR+TTS）集成快，但依赖 **外网质量、资费与数据出境策略**。本系统 **本地优先 + 云端兜底** 在 **校园专网环境** 下更易演示，论文中可强调 **工程权衡** 而非「本地一定更强」。

## 2.7 可追溯与审计相关技术

### 2.7.1 设计目标

可追溯性在本系统中体现为：**智能体是否调用工具、调用何工具、关键操作是否留痕**，并与 **业务库** 区分层次——**在线业务**、**对话与工具快照**、**审计日志** 各司其职（参见 **`docs/数据库设计（AI与可追溯）.md`** 中智能与审计域划分）。

### 2.7.2 数据层技术要点

| 机制 | 技术含义 | 与「仅打文件日志」对比 |
|------|----------|-------------------------|
| **`ai_conversations` / `ai_messages`** | 会话与逐条消息落库；`tool_calls`、`suggestions` 等 JSON 字段保留 **结构化证据** | 文件日志难关联用户与会话、不便做运营查询 |
| **`audit_logs`** | 记录操作者、动作、对象与 JSON 详情；AI 服务可在对话结束后 **异步 best-effort 写入** | 满足 **责任追溯** 与答辩演示「可查库」 |
| **`post_vector_index` 与帖子分离** | 索引与正文解耦，便于 **换模型重建** | 避免宽表膨胀、职责清晰 |

**选用理由**：论文若强调 **AI 治理**，仅有 Prompt 约束不够；**落库** 的 `tool_calls` 与 `audit_logs` 提供 **可查询、可截图、可对照需求** 的证据链，与 **2.4** 中 Guardrails（工具白名单、移动端数据域闭包）形成 **「约束 + 留痕」** 闭环。

### 2.7.3 与「仅依赖提示词安全」的对比

提示词层可声明「不得越权」，但 **无法替代执行路径裁剪**。本系统在 **工具选择阶段** 按 `clientType`、`role`、`userId` 计算允许工具列表，无权限则不编译 Agent 图或返回降级文案——论文可引用 **`docs/LangChain设计与实现（ai-service）.md`** 中 Guardrails 小节作为设计依据。

### 2.7.4 与本章前文的小结对照表

将 **2.3～2.6** 及本节 **2.7.1～2.7.3** 已分散叙述的选型 **压缩为一览**，便于本章结尾或答辩总述。

| 领域 | 选用技术 | 主要对比对象 | 核心选用理由（一句话） |
|------|----------|--------------|------------------------|
| Web 与 ASGI | FastAPI + Uvicorn | Flask 等 | 强类型 API、异步友好、与 AI 示例生态一致 |
| Agent 编排 | LangChain + LangGraph | 裸 SDK、多智能体框架 | 工具协议 + 显式状态图，便于工程化叙述 |
| 检索 | 向量 + MySQL 索引表 | 纯关键词、专用向量库 | 语义召回与现有运维栈平衡，可演进 |
| 语音 | Whisper + Coqui（主） | 纯云端 | 本地演示与兜底可用性兼顾 |
| 追溯 | 关系库落库 + 审计表 | 仅应用日志 | 可查询、可解释；与 Guardrails 形成约束+留痕 |

**与《数据库设计（AI与可追溯）》的分工**：数据库章写 **表结构、外键与域划分**；本节 **2.7** 只从 **「采用何种技术手段实现可追溯」** 角度写，避免第二章与数据库章在字段级设计上重复。

## 使用说明

- **答辩口语**：「Python 这边 FastAPI 提供接口，LangGraph 管 Agent 和工具循环，检索向量存在 MySQL 的索引表里由 Java 查，语音本地 Whisper/Coqui 为主，对话和工具调用会落库、关键行为写审计表。」  
- **避免夸大**：多智能体、全链路流式、重排器（Reranker）等若未完整实现，放在 **展望** 或 **局限**，与本篇「已实现技术」区分。  
- **合订后本章小结**：可增加一句——**智能能力由独立服务承载，与 Spring Boot 业务库解耦；通过工具白名单与审计落库形成可追溯闭环。**
